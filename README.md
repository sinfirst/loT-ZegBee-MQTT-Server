# IoT Device Management Server (Golang)

Серверная часть системы мониторинга и охраны имущества с использованием ZigBee‑датчиков. Проект предназначен для арендуемого жилья: датчики вибрации, наклона и падения, установленные на крупных предметах (телевизор, мебель, техника), через хаб на базе Raspberry Pi отправляют данные на сервер. Сервер обрабатывает события, сохраняет их в базу данных и уведомляет владельца через Telegram.

Реализация выполнена на **Go** с использованием асинхронной обработки, MQTT для связи с хабами и REST API для управления устройствами и пользователями.

## Возможности

- Регистрация пользователей по Telegram ID.
- Привязка ZigBee‑хаба (Raspberry Pi) и его датчиков к пользователю.
- Приём MQTT‑сообщений от хабов (статусы устройств, события, уровень батареи).
- Хранение истории событий (движение, вибрация, наклон, падение).
- Отправка уведомлений владельцу через Telegram при важных событиях (низкий заряд батареи, срабатывание датчика).
- REST API для управления устройствами и просмотра событий.
- Аутентификация через JWT (пользователи идентифицируются по Telegram ID).

## Архитектура

Система состоит из следующих компонентов:

- **ZigBee датчики** – устанавливаются на объекты, передают данные по ZigBee.
- **Хаб (Raspberry Pi + Sonoff ZigBee USB Dongle Plus)** – собирает данные с датчиков и публикует их в MQTT брокер.
- **MQTT брокер** (Mosquitto) – обеспечивает обмен сообщениями между хабом и сервером.
- **Сервер (данный проект)** – подписывается на MQTT топики, обрабатывает входящие сообщения, сохраняет в БД, отправляет уведомления и предоставляет REST API.
- **База данных** – PostgreSQL для хранения пользователей, устройств, событий, уведомлений.
- **Telegram Bot** – отправляет уведомления пользователям.

```
[ZigBee sensors] <--ZigBee--> [Hub (RPi)] <--MQTT--> [MQTT Broker] <--MQTT--> [Go Server]
                                                                   |
                                                              [PostgreSQL]
                                                                   |
                                                              [Telegram Bot API] --> [User's Telegram]
```

## Технологии

- **Go** 1.21+ – основной язык разработки.
- **MQTT** – библиотека [Mosquitto](https://github.com/eclipse-mosquitto/mosquitto) для работы с MQTT.
- **PostgreSQL** – реляционная БД.
- **Миграции** – [goose](https://github.com/pressly/goose).
- **HTTP-роутер** – [chi](https://github.com/go-chi/chi).
- **Логирование** – [zap](https://go.uber.org/zap).
- **JWT** – [golang-jwt/jwt](https://github.com/golang-jwt/jwt) для аутентификации.
- **Конфигурация** – YAML + переменные окружения.
- **Тестирование** – стандартный пакет `testing`, [testify](https://github.com/stretchr/testify).

## Конфигурация

Настройки сервера задаются через файл `config.yaml` и/или переменные окружения (с приоритетом переменных окружения). Пример `config.yaml`:

```yaml
server:
  port: 8080
  jwt_secret: "your-secret-key"

database:
  dsn: "postgres://user:pass@localhost:5432/iot?sslmode=disable"

mqtt:
  broker: "tcp://localhost:1883"
  client_id: "iot-server"
  topics:
    - "home/+/+/status"
    - "home/+/+/battery"
    - "home/+/+/movement"

telegram:
  bot_token: "YOUR_BOT_TOKEN"
  chat_id: ""  # можно задать глобально или получать из БД

log_level: "info"
```

Переменные окружения переопределяют соответствующие поля (например, `SERVER_PORT`, `DATABASE_DSN`).

## API

Все эндпоинты, кроме регистрации, требуют аутентификации. Токен передаётся в заголовке `Authorization: Bearer <JWT>`. JWT выдаётся после регистрации пользователя (эндпоинт `POST /api/user/register` возвращает токен).

### 1. Регистрация/получение пользователя

**POST** `/api/user/register`

Регистрирует нового пользователя по Telegram ID или возвращает существующего.

**Запрос:**
```json
{
  "telegram_id": 123456789,
  "username": "john_doe"
}
```

**Успешный ответ (200):**
```json
{
  "status": "ok",
  "user_id": "uuid-пользователя",
  "message": "User registered successfully",
  "token": "jwt-token"           // токен для последующих запросов
}
```

**Ошибки:** 400 (неверный запрос), 409 (уже существует, но возвращается тот же пользователь с токеном), 500.

### 2. Регистрация устройства (хаба)

**POST** `/api/hub/register`

Привязывает хаб и его устройство к пользователю.

**Запрос:**
```json
{
  "user_id": "uuid-пользователя",
  "device_id": "unique-device-id"
}
```

**Успешный ответ (200):**
```json
{
  "status": "ok",
  "devices_id": "unique-device-id",
  "message": "Device registered"
}
```

Если устройство уже привязано к этому же пользователю, возвращается:
```json
{
  "status": "ok",
  "message": "User already registered in this hub. Check api/user/devices."
}
```

**Ошибки:** 400 (отсутствуют поля), 404 (пользователь не найден), 409 (устройство уже принадлежит другому пользователю), 500.

### 3. Получение списка устройств пользователя

**GET** `/api/user/{user_id}/devices`

Возвращает все устройства, привязанные к указанному пользователю.

**Параметры пути:** `user_id` – UUID пользователя.

**Успешный ответ (200):**
```json
{
  "status": "ok",
  "devices": [
    {
      "device_id": "unique-device-id",
      "ieee_addr": "00:11:22:33:44:55:66:77",
      "user_id": "uuid-пользователя",
      "hub_id": "hub-identifier",
      "model_id": "model-name",
      "device_type": "vibration",
      "device_online": true,
      "battery_percentage": 85,
      "last_seen": "2024-01-15T10:30:00Z",
      "last_event": "movement",
      "link_quality": 200
    }
  ]
}
```

**Ошибки:** 400 (неверный user_id), 404 (пользователь не найден), 500.

### 4. Получение информации об устройстве

**GET** `/api/device/{device_id}`

Детальная информация по конкретному устройству.

**Параметры пути:** `device_id` – идентификатор устройства.

**Успешный ответ (200):**
```json
{
  "status": "ok",
  "device": {
    "device_id": "...",
    "ieee_addr": "...",
    "user_id": "...",
    "hub_id": "...",
    "model_id": "...",
    "device_type": "vibration",
    "device_online": true,
    "battery_percentage": 85,
    "last_seen": "...",
    "last_event": "movement",
    "link_quality": 200
  }
}
```

**Ошибки:** 404 (устройство не найдено), 500.

### 5. Удаление устройства

**DELETE** `/api/hub/{device_id}`

Отвязывает устройство от пользователя (удаляет из системы).

**Параметры пути:** `device_id` – идентификатор устройства.

**Успешный ответ (200):**
```json
{
  "status": "ok",
  "deleted_devices": ["device_id1", "device_id2"],
  "message": "Devices deleted"
}
```

**Ошибки:** 404 (устройство не найдено), 500.

### 6. Получение событий пользователя

**GET** `/api/user/{user_id}/events`

События со всех устройств пользователя за последние N часов.

**Параметры пути:** `user_id` – UUID пользователя.  
**Query-параметр:** `hours` (целое) – количество часов для выборки.

**Успешный ответ (200):**
```json
{
  "status": "ok",
  "events": [
    {
      "id": "event-id",
      "hub_id": "hub-id",
      "device_id": "device-id",
      "event_type": "movement_detection",
      "link_quality": 200,
      "created_at": "2024-01-15T10:30:00Z"
    }
  ]
}
```

**Ошибки:** 400 (неверный hours), 404 (пользователь не найден), 500.

### 7. Получение событий устройства

**GET** `/api/device/{device_id}/events`

События конкретного устройства за последние N часов.

**Параметры пути:** `device_id` – идентификатор устройства.  
**Query-параметр:** `hours` (целое).

**Успешный ответ:** аналогичен предыдущему.

**Ошибки:** 400 (неверный hours), 404 (устройство не найдено), 500.

## MQTT взаимодействие

Сервер подписывается на топики вида:

- `home/+/+/status` – получение текущего статуса устройства (онлайн/офлайн, уровень сигнала).
- `home/+/+/battery` – уровень заряда батареи.
- `home/+/+/movement` – события движения/вибрации/наклона.

Плюс (`+`) – это wildcard для комнаты и имени устройства. Пример полного топика: `home/livingroom/tv/status`.

Сообщения от хаба приходят в формате JSON (или простого текста). Пакет `internal/mqtt` парсит их, определяет тип события, сохраняет в БД и при необходимости инициирует отправку уведомления через `internal/notification`.

## Уведомления

Уведомления отправляются через Telegram-бота. При регистрации пользователя указывается его `telegram_id`, на который бот может отправлять сообщения.

Уведомления генерируются при:

- Низком заряде батареи устройства (<20%).
- Срабатывании датчика (движение, вибрация, падение).
- Потере связи с устройством (если не было событий более заданного времени).

Для асинхронной отправки используется горутина или очередь задач (в перспективе можно заменить на более мощный брокер, но пока достаточно простого фонового процесса).

## Запуск и развёртывание

### Требования

- Go 1.21+
- PostgreSQL 12+
- MQTT брокер (например, Mosquitto)

### Локальный запуск

1. Клонируйте репозиторий:
   ```bash
   git clone https://github.com/yourname/iot-server.git
   cd iot-server
   ```

2. Создайте базу данных и выполните миграции:
   ```bash
   createdb iot
   migrate -path internal/storage/migrations -database "postgres://localhost/iot?sslmode=disable" up
   ```

3. Настройте конфигурацию: скопируйте `config.yaml.example` в `config.yaml` и укажите свои параметры (БД, MQTT, Telegram токен).

4. Соберите и запустите:
   ```bash
   go build -o server cmd/server/main.go
   ./server
   ```

Или используйте `go run`:
```bash
go run cmd/server/main.go
```


## Тестирование

Юнит-тесты написаны с использованием стандартного пакета `testing` и `testify`. Для тестирования работы с БД используется отдельная тестовая БД или моки.

Запуск всех тестов:
```bash
go test ./... -cover
```

Для интеграционного тестирования MQTT-части можно поднять тестовый брокер (например, в контейнере).
